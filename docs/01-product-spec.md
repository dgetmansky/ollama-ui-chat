# Product Spec: Ollama UI GDP

## 1. Назначение

`ollama-ui-gdp` - это локальный web UI для работы с удаленным или локальным Ollama API в режиме прикладного тестирования и диагностики.

Приложение не пытается быть "универсальным красивым чат-клиентом". Его основная цель - дать быстрый и минималистичный интерфейс для:

- выбора модели;
- отправки запросов в Ollama;
- сравнения поведения `/api/generate` и `/api/chat`;
- просмотра raw JSON запросов и ответов;
- просмотра серверных метрик и вычисленных производных метрик;
- сохранения локальных сессий в файловой системе.

## 2. Контекст использования

Основной сценарий использования:

- Ollama API работает на другом сервере;
- пользователь пробрасывает порт локально через SSH tunnel;
- UI запускается локально на машине пользователя;
- история работы хранится локально в проекте, без внешней БД и без синхронизации между машинами.

Приложение должно одинаково хорошо работать и с локальным Ollama, и с удаленным Ollama через tunnel.

## 3. Принципы продукта

- Минимализм: без лишней навигации, сложных экранов и необязательной "продуктовой" логики.
- Наблюдаемость: пользователь должен видеть не только текст ответа, но и то, что реально ушло в API и что реально вернулось.
- Сравнимость: UI должен помогать сравнивать модели, параметры и endpoint'ы.
- Локальность: сессии и служебные данные хранятся только в проекте.
- Человекочитаемость: сессии сохраняются в JSON.

## 4. Целевые сценарии

### 4.1. Базовый запрос к модели

Пользователь:

- открывает UI;
- выбирает endpoint;
- выбирает модель;
- задает параметры генерации;
- отправляет запрос;
- получает ответ;
- смотрит текст, raw JSON и метрики.

### 4.2. Сравнение endpoint'ов

Пользователь отправляет сходные запросы через `/api/generate` и `/api/chat`, чтобы понять разницу в поведении, формате ответа, стриминге и метриках.

### 4.3. Сравнение моделей

Пользователь меняет модель и повторяет запрос, чтобы сравнить качество ответа, скорость и статистику.

### 4.4. Диагностика удаленного backend

Пользователь проверяет доступность endpoint'а, обновляет список моделей, смотрит сырые ответы и метрики, чтобы понять, жив ли backend и что именно он делает.

### 4.5. Продолжение локальной сессии

Пользователь позже возвращается к ранее сохраненной сессии, открывает JSON-сессию из папки `sessions/` через UI и продолжает работу.

## 5. Обязательный функционал MVP / v1

### 5.1. Работа с Ollama

Приложение должно поддерживать:

- endpoint `/api/generate`;
- endpoint `/api/chat`;
- выбор активного endpoint'а в UI;
- выбор модели через `/api/tags`;
- обновление списка моделей без перезапуска приложения;
- проверку доступности backend через `Ping`.

### 5.2. Чат и запросы

Приложение должно поддерживать:

- ввод пользовательского сообщения;
- отправку запроса;
- показ ответа модели;
- режим `stream on/off`;
- остановку streaming-ответа через `Stop / Abort`;
- сохранение истории сообщений внутри сессии.

### 5.3. Параметры запроса

В `v1` UI должен позволять задавать:

- `num_predict`;
- `temperature`.

Параметры должны попадать в отправляемый payload и сохраняться в сессии.

### 5.4. Диагностические блоки

После каждого запроса UI должен показывать:

- `Last request payload`;
- `Last response raw JSON`;
- `Stats`;
- derived metrics.

Под `Stats` понимаются прежде всего поля ответа Ollama, связанные с длительностью и токенами, например:

- `total_duration`;
- `load_duration`;
- `prompt_eval_count`;
- `prompt_eval_duration`;
- `eval_count`;
- `eval_duration`.

Под derived metrics понимаются вычисляемые на стороне приложения показатели, например:

- total sec;
- load sec;
- prompt tokens/sec;
- eval tokens/sec.

### 5.5. Локальные сессии

Приложение должно:

- хранить сессии в папке `sessions/`;
- хранить каждую сессию отдельным JSON-файлом;
- автоматически сохранять изменения сессии после запроса;
- показывать список локальных сессий;
- позволять открыть существующую сессию;
- позволять создать новую сессию;
- позволять удалить сессию.

Имя файла сессии должно строиться по шаблону:

`YYYY-MM-DDTHH-MM-SS-<uuid>.json`

В `v1` отдельное пользовательское имя сессии не требуется.

## 6. UX-требования v1

- Интерфейс должен быть одностраничным.
- Интерфейс должен быть минималистичным и утилитарным.
- Главный рабочий цикл должен быть быстрым: выбрать модель, отправить запрос, посмотреть ответ и диагностику.
- Пользователь не должен терять данные сессии при сетевой ошибке или недоступности Ollama.
- Ошибки должны показываться явно и понятно.

## 7. Ограничения v1

- Без пользовательских аккаунтов.
- Без удаленного хранения истории.
- Без синхронизации между машинами.
- Без БД.
- Без сложной многопользовательской логики.
- Без сложной маршрутизации по страницам.

## 8. Что откладывается в v2+

Следующие идеи считаются полезными, но не входят в обязательный объем `v1`:

- дополнительные параметры, например `top_p`, `seed`, `top_k`, `num_ctx`, `repeat_penalty`, `stop`;
- preset-профили, например `Fast`, `Balanced`, `Concise`;
- отдельный режим показа raw chunks / thinking traces;
- расширенный журнал всех вызовов, а не только последний диагностический снимок;
- более богатое сравнение результатов между моделями и endpoint'ами;
- экспорт и импорт сессий;
- расширенная фильтрация и поиск по сессиям;
- более сложные визуализации метрик.

## 9. Критерии готовности v1

Версию `v1` можно считать готовой, если:

- UI запускается локально как web-приложение;
- пользователь может выбрать `/api/generate` и `/api/chat`;
- пользователь может обновить список моделей;
- пользователь может отправить запрос и получить ответ;
- `stream on/off` работает;
- `Stop / Abort` работает для streaming;
- после запроса показываются request payload, raw response, stats и derived metrics;
- сессия сохраняется в `sessions/*.json`;
- пользователь может открыть ранее сохраненную сессию;
- приложение корректно работает с локальным Ollama или с Ollama через SSH tunnel.
