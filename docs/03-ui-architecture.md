# UI Architecture: Ollama UI GDP

## 1. Цель документа

Этот документ описывает структуру интерфейса `v1`, распределение ответственности между частями экрана и ключевые пользовательские состояния.

## 2. Общий принцип интерфейса

Интерфейс `v1` должен быть:

- одностраничным;
- минималистичным;
- ориентированным на быстрый цикл тестирования;
- утилитарным, а не декоративным.

Главный смысл интерфейса - не "красиво чатиться", а быстро сравнивать режимы работы Ollama и видеть диагностические данные.

## 3. Основные зоны интерфейса

### 3.1. Панель управления

Панель управления располагается в верхней части экрана и содержит:

- переключатель endpoint (`/api/generate` или `/api/chat`);
- выбор модели;
- переключатель `stream on/off`;
- поле `num_predict`;
- поле `temperature`;
- кнопку `Refresh models`;
- кнопку `Ping`;
- кнопку `New session`.

Назначение панели - собрать все управляющие параметры одного запроса в одном месте, без необходимости ходить по отдельным экранам.

### 3.2. Список сессий

Список сессий отображает локальные JSON-сессии из папки `sessions/`.

Для каждой сессии в `v1` достаточно показывать:

- дату создания;
- время создания;
- короткий идентификатор, если это нужно для различения.

В `v1` не требуется:

- теги;
- поиск;
- группировка;
- ручные названия;
- сложная сортировка кроме естественной сортировки по времени.

### 3.3. Основная область чата

Основная область содержит:

- историю сообщений текущей сессии;
- текстовое поле ввода;
- кнопку `Send`;
- кнопку `Stop` во время активного streaming-запроса.

Здесь пользователь выполняет основной сценарий: отправляет запрос и читает текстовый результат.

### 3.4. Диагностическая панель

Диагностическая панель показывает:

- `Last request payload`;
- `Last response raw JSON`;
- `Stats`;
- `Derived metrics`.

В `v1` панель должна быть постоянно доступна на том же экране, без перехода на отдельную страницу.

## 4. Предлагаемая компоновка

Для `v1` рекомендуется следующая логика экрана:

- сверху: панель управления;
- слева или в узкой колонке: список сессий;
- в центре: чат;
- справа или в нижней панели: диагностика.

Если размер экрана ограничен:

- диагностическая панель может уходить вниз под чат;
- список сессий может сворачиваться;
- приоритет всегда остается за доступностью чата и панели управления.

## 5. Пользовательские действия

### 5.1. Запуск приложения

При старте UI:

- запрашивает список моделей;
- запрашивает список локальных сессий;
- открывает либо последнюю сессию, либо пустое состояние новой сессии;
- показывает доступность backend, если это можно сделать без лишней задержки.

### 5.2. Создание новой сессии

По кнопке `New session` приложение:

- создает новую пустую сессию;
- назначает ей идентификатор и имя файла по шаблону;
- сохраняет ее в `sessions/`;
- переключает UI на эту сессию.

### 5.3. Открытие существующей сессии

При выборе сессии из списка UI:

- загружает содержимое JSON;
- восстанавливает историю сообщений;
- восстанавливает выбранный endpoint;
- восстанавливает модель;
- восстанавливает `stream`;
- восстанавливает сохраненные request options;
- показывает последние диагностические данные, если они есть.

### 5.4. Отправка запроса

При отправке запроса UI:

- берет текущее состояние управляющей панели;
- отправляет его на локальный backend;
- показывает процесс выполнения;
- обновляет историю сообщений;
- обновляет диагностические блоки;
- сохраняет сессию.

### 5.5. Прерывание запроса

Во время streaming-запроса UI:

- показывает кнопку `Stop`;
- после прерывания сохраняет уже полученный частичный результат;
- не скрывает факт, что ответ был прерван.

## 6. Состояния интерфейса

В `v1` UI должен явно поддерживать следующие состояния:

- пустая новая сессия;
- загрузка моделей;
- загрузка сессии;
- выполнение запроса;
- streaming-ответ;
- успешно завершенный запрос;
- прерванный запрос;
- ошибка backend Ollama;
- ошибка локального backend;
- ошибка чтения или записи сессии.

Эти состояния должны отличаться визуально и текстово, чтобы пользователь понимал, что именно происходит.

## 7. Правила минимализма

Чтобы сохранить утилитарный характер `v1`, интерфейс не должен включать:

- маркетинговые элементы;
- лишнюю декоративность;
- сложный wizard;
- многоуровневую навигацию;
- перегруженные модальные окна;
- функции, которые не помогают отправить запрос, посмотреть результат или восстановить сессию.

## 8. Приоритет информации на экране

При конфликте места на экране приоритет должен быть таким:

1. Панель управления.
2. Основная область чата.
3. Диагностическая панель.
4. Список сессий.

Это отражает основной рабочий цикл: сначала управлять запросом, потом видеть ответ, потом смотреть диагностику, и только затем переключать сессии.

## 9. Договоренности для v1

- Один экран без роутинга.
- Две API-схемы: `/api/generate` и `/api/chat`.
- Две ручные настройки генерации: `num_predict` и `temperature`.
- Локальные JSON-сессии в `sessions/`.
- Диагностические панели видимы в рамках основного рабочего экрана.

## 10. Что осознанно не делаем в v1

- визуальный редактор payload;
- ручное редактирование raw JSON прямо в UI;
- встроенное сравнение "два ответа рядом";
- сохраненные пользовательские пресеты;
- сложные фильтры по сессиям;
- отдельный экран аналитики;
- поддержку множества backend'ов одновременно.
